# 🚀 Azure Bicep Notes — Bicep Format & Concepts Explained

---

# 📘 What is Bicep?

Bicep is an Infrastructure as Code (IaC) language used to deploy Azure resources.

Using Bicep, we define Azure infrastructure in code format.

Example:

- Virtual Machines
- Storage Accounts
- Web Apps
- VNets
- Databases

---

# 🔥 What Does "Bicep is Declarative" Mean?

Bicep is a **declarative language**.

This means:

✅ We tell Azure **WHAT** we want  
❌ We do NOT tell Azure **HOW** to create it step-by-step

---

# 🧠 Real-Life Example

## 🎂 Declarative Style

You say:

```text
I want a birthday cake.
```

You do NOT explain:

```text
1. Buy flour
2. Add sugar
3. Bake cake
4. Add cream
```

You only describe the FINAL RESULT.

Azure handles the internal process automatically.

---

# ⚔️ Imperative vs Declarative

| Imperative | Declarative |
|---|---|
| Step-by-step instructions | Final desired state |
| User controls flow | Azure controls flow |
| Complex | Simpler |
| Manual dependency handling | Automatic dependency handling |

---

# 🔥 Important Concept

## Order Does NOT Matter in Bicep

In Bicep:

✅ Elements can appear in ANY order

Because Azure automatically understands dependencies.

# 🏗️ Bicep File Structure

A Bicep file contains multiple elements.

---

# 1️⃣ Metadata

Used for extra information/documentation.

```bicep
metadata description = 'Creates a storage account'
```

---

## Purpose

- Documentation
- Description
- Better readability

---

# 2️⃣ targetScope

Defines where deployment will happen.

```bicep
targetScope = 'resourceGroup'
```

---

## Available Scopes

| Scope | Meaning |
|---|---|
| resourceGroup | Deploy inside Resource Group |
| subscription | Deploy at Subscription level |
| managementGroup | Deploy at Management Group |
| tenant | Deploy at Tenant level |

---

# 3️⃣ Parameters (`param`)

Used for user input.

```bicep
param location string
```

---

## Purpose

Allows reusable deployments.

Example:

```text
Location = Central India
Environment = Production
```

---

# 🎯 Parameter with Default Value

```bicep
param sku string = 'Standard_LRS'
```

If user does not provide value, default value is used.

---

# 🎯 Parameter Validation

```bicep
@minLength(3)
@maxLength(11)
param storagePrefix string
```

---

## Meaning

| Decorator | Purpose |
|---|---|
| @minLength | Minimum characters |
| @maxLength | Maximum characters |

---

# 4️⃣ Variables (`var`)

Used to store reusable values.

```bicep
var env = 'prod'
```

---

# Example

```bicep
var uniqueStorageName = '${storagePrefix}${uniqueString(resourceGroup().id)}'
```

# 5️⃣ Resource

Used to create actual Azure resources.

```bicep

}
```

---

# 🔥 Resource Breakdown


|---|---|

# 🏗️ Resource Example

```bicep
resource stg 'Microsoft.Storage/storageAccounts@2025-06-01' = {
  name: uniqueStorageName
  location: location
```

---

# 🔍 Explanation

| Property | Meaning |
|---|---|

| location | Azure region |
| sku | Storage performance type |
| kind | Storage type |
| supportsHttpsTrafficOnly | Only HTTPS traffic allowed |

# 🔒 Security Best Practice

# 6️⃣ Modules

Modules are reusable Bicep files.

```bicep
module webModule './webApp.bicep' = {
}
```

---

# 🎯 Purpose of Modules

Modules help in:

✅ Reusability  
✅ Cleaner code  
✅ Team collaboration  
✅ Large project management

---

# 📦 Module Example

```bicep
module webModule './webApp.bicep' = {
  name: 'webDeploy'

  params: {
    skuName: 'S1'
    location: location
  }
}
```

---

# 🔍 Explanation

| Part | Meaning |
|---|---|
| module | Calling another Bicep file |
| webModule | Symbolic name |
| ./webApp.bicep | Path of Bicep file |
| params | Passing values to module |

---

# 🧠 Real-Life Module Example

Imagine:

```text
main.bicep
```

Creates:

- Storage
- Network

Then calls:

```text
webApp.bicep
```

Which creates:

- Web App
- App Service Plan

This makes infrastructure modular.
